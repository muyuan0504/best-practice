# 迭代器与可迭代协议 — 问题与解释

## 1. for...of 打印的 item 为什么不是对应的 value？

**现象**：自定义迭代器在 `for (const item of obj)` 中，`item` 没有按预期得到 `next()` 返回的 `value`，或循环体根本没有执行。

**原因**：根据迭代协议，只有当 `next()` 返回 **`done: false`** 时，`for...of` 才会把本次的 `value` 赋给 `item` 并执行一次循环体。若**第一次**就返回 `done: true`，引擎会认为“没有可迭代项”，循环体一次都不会执行，因此看不到任何打印。

**正确写法**：要 yield 的值必须用 `{ value: xxx, done: false }` 返回；只在迭代结束时再返回 `{ value: undefined, done: true }`。

---

## 2. 如果第一次就返回 done 为 true，返回的 item 会包含 value 吗？

**结论**：不会。而且严格说，此时**根本没有“这一轮的 item”**。

- `for...of` 只有在某次 `next()` 返回 **`done: false`** 时，才会把该次的 **`value`** 赋给 `item` 并执行循环体。
- 第一次就返回 `done: true` 时，循环被视为已结束，**循环体一次都不会执行**，所以不会有“这一轮的 item”。
- `next()` 返回的对象 `{ value: 1, done: true }` 本身有 `value`，但赋给 `item` 的始终是“某次 `done: false` 时的 `value`”；当 `done: true` 时循环直接结束，不会再用这次返回值给 `item` 赋值。

---

## 3. console.log 不打印，是因为循环没有进入吗？

**结论**：是的。

当**第一次**调用 `next()` 就得到 `{ value: xxx, done: true }` 时，引擎认为迭代已结束，**循环体一次都不会执行**，相当于 `for...of` 从未“进入”循环，所以循环内的 `console.log` 不会运行。

可以理解为：只有 `done === false` 时才会把 `value` 赋给 `item` 并执行循环体；一上来就是 `done: true`，就相当于“没有下一项”，循环体一次都不会跑。

---

## 4. `const [one, another] = random` 是怎么实现的？

**现象**：`random` 是一个迭代器从不返回 `done: true` 的无限序列，但 `const [one, another] = random` 能正常得到两个随机数且不卡死。

**原因**：对可迭代对象做**数组解构**时，会按顺序从迭代器里“取”值，**取够解构里的变量个数就停**，不会一直跑到 `done: true`。

**等价过程**：

1. 调用 `random[Symbol.iterator]()` 得到迭代器。
2. 需要几个值就调用几次 `next()`，依次把 `value` 赋给左边的变量。
3. 取够（这里是 2 个）就停止，不再调用 `next()`。

等价于：

```js
const iterator = random[Symbol.iterator]()
const one     = iterator.next().value   // 第 1 次
const another = iterator.next().value   // 第 2 次
```

**总结**：数组解构在右侧是 iterable 时，会按顺序消费迭代器，取值的个数 = 左边变量个数，取够就停，因此即使迭代器是“无限”的也不会死循环。
