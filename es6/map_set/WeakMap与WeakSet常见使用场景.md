# WeakMap 与 WeakSet：常见使用场景

二者共同点：**键（WeakMap）或值（WeakSet）必须是对象**；**不可迭代**；集合对成员的引用是**弱引用**——当对象在别处不再被引用时，可被垃圾回收，条目会从 WeakMap / WeakSet 中消失，从而**避免“挂着字典不放”导致的内存泄漏**。

下面按场景说明什么时候值得用它们，而不是普通的 `Map` / `Set`。

---

## WeakMap 常见场景

### 1. 给 DOM 节点或第三方对象挂「私有」数据

需要在不改变对象本身结构的前提下存元数据（状态、缓存、配置）。用普通 `Map` 会以强引用握住节点，节点已从文档移除且脚本不再引用时，仍可能无法回收。  
**WeakMap 以对象为键**，脚本放开对该节点的唯一引用后，元数据会随对象一起可被回收。

典型思路：`weakMap.set(domNode, { ... })`，组件卸载或节点移除后不再保留别处引用即可。

### 2. 实例关联数据（库内部「侧表」）

框架或库需要「每个实例一份」扩展数据，又不想把字段写在实例上（污染原型 / 暴露内部字段）。可以用 **WeakMap：实例对象 → 内部状态**。  
对外只暴露 API，真正的状态在 WeakMap 里；实例被销毁后，侧表条目不会阻止回收。

（现代 JS 也可用类私有字段 `#x`；WeakMap 仍适合「键不是 class 实例」或需要在多个模块间共享同一张侧表而不导出字段名的场合。）

### 3. 与对象生命周期一致的缓存（memo）

对「某个对象」的计算结果缓存：`weakMap.set(obj, computed)`。当 `obj` 不再被使用时，缓存条目不应再占用内存。  
若用 `Map` 做 `obj → 缓存`，必须记得在适当时机 `delete`，否则容易泄漏。**WeakMap** 把生命周期交给 GC，适合**缓存条目必须随对象消亡**的模式。

注意：无法遍历全部缓存键，因此不能做「全局 LRU 淘汰」这类依赖枚举的策略；只适合「按对象存活自然失效」的缓存。

### 4. 避免循环引用时的辅助映射

在序列化、克隆、响应式系统等场景中，有时需要 `obj → 代理或副本` 的映射。若映射用强引用且图很大，结束处理后仍可能延缓回收。用 WeakMap 可降低长期占用风险（仍要结合业务在合适的时机断开其它引用）。

## WeakSet 常见场景

### 1. 「标记」一类对象，且不延长其寿命

需要维护「这些对象已被注册 / 已处理 / 属于某内部集合」，但**不希望**因为这个集合的存在而阻止对象被 GC。  
例如：插件注册了若干实例，卸载时应允许实例消失；用 **WeakSet.add(obj)** 表示 membership，不用后别处不再引用 `obj` 即可回收。

### 2. 库内部的「品牌检测」（brand checking）

判断某个对象是否由本库创建：可用 WeakSet 存放「认证过的」对象引用，`weakSet.has(obj)`。  
不把标记写在对象属性上，避免与用户字段冲突；对象销毁后集合里也不会强留住它。

### 3. 临时去重（仅限「对象引用」维度）

若要去重的是**对象引用**（同一引用才算重复），且集合不应阻止这些对象被回收，可用 WeakSet。  
若要去重的是原始值（字符串、数字等），必须用 `Set`，WeakSet 不支持。

---

## 何时不要用 WeakMap / WeakSet

- 需要 **`for...of`、`.size`、批量清空、枚举全部键** → 用 `Map` / `Set`。
- 键或成员是 **字符串、数字、Symbol** 等原始类型 → 不能用 WeakMap（键）/ WeakSet（值），只能用普通 Map/Set。
- 需要 **稳定、可预测的缓存大小与淘汰策略** → WeakMap 不可枚举，不适合单独实现 LRU；通常用 Map + 显式淘汰或专用缓存结构。

---

## 一句话对照

| 结构 | 适合解决的问题 |
|------|----------------|
| **WeakMap** | 「对象 → 附属数据」，生命周期跟随对象，避免泄漏 |
| **WeakSet** | 「某对象是否在里面」的成员标记，不阻止对象被 GC |

两者都是：**在「只要对象还活着就需要这张表」的前提下，对象死了就让表里的痕迹一起消失**。
